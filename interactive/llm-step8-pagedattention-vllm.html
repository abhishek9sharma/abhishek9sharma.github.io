PagedAttention & vLLM: managing the cache

Step 7 showed the KV cache is essential but enormous — and its size per request grows and shrinks unpredictably. Store it naively and you waste 60–80% of GPU memory. PagedAttention fixes this by borrowing the oldest trick in operating systems: paging. It's the core idea behind vLLM.

Builds on Step 7 (the KV cache and its memory cost). Self-contained — the key facts are recapped as we go.
01The Waste

Why contiguous allocation bleeds memory

The naive approach: when a request arrives, reserve one contiguous block of memory big enough for the maximum sequence length it might reach — say 2048 tokens — even though it might only generate 30. This creates two kinds of waste:

First — what is a "request"?

A request is one user's call to the model: their prompt plus the tokens being generated back to them. A busy GPU serves many at once, and each one needs its own slice of KV-cache memory (the keys & values from Step 7). Throughout this page, R1–R4 are four such concurrent requests — each a different user, prompt, and length:

R1"Translate this email into French…" · short
R2"Write a haiku about the ocean…" · tiny
R3"Summarize this 5-page report…" · long
R4"def quicksort(arr): …" · medium

Each finishes at a different, unpredictable time — which is exactly what makes packing them into GPU memory hard.

Naive contiguous KV allocation — 4 requests on a GPU

— of reserved KV memory actually holds real tokens.
The rest is reserved-but-empty (internal fragmentation) or unusable gaps (external fragmentation).
Two fragmentation types. Internal: each request reserves its max length but uses far less — the reserved-but-empty tail is dead memory for the request's whole lifetime. External: when a request finishes and frees its block, the gap may be too small or oddly-shaped for the next request's contiguous need. Real systems lose 60–80% of KV memory this way.
02The Idea

Paging: store the cache in small blocks, anywhere

Operating systems solved this for RAM decades ago. Don't demand one big contiguous chunk — split memory into fixed-size blocks (vLLM uses 16 tokens each) and let a sequence's blocks live anywhere, in any order. A block table remembers where each one is. Toggle the two schemes and watch utilization jump:

—

New tokens are the whole reason paging wins. During generation the model appends one token at a time to a request's KV cache (Step 7). With paging, a request holds only the blocks it has filled so far; when its last block fills up (every 16th token) it grabs one fresh block from the free pool — on demand. The naive scheme, by contrast, reserves all 2048 tokens up front on token 1. Hit “R1 generates a token” above and watch a new block appear exactly when R1 crosses a 16-token boundary.

The block table — logical tokens → physical blocks

Just like an OS page table. The sequence thinks its tokens are contiguous (positions 0–47); physically they're scattered wherever there was a free block.

The waste collapses to almost nothing. The only unused space is the partially-filled last block of each sequence — at most 15 tokens with a block size of 16. That brings total waste to under 4%, versus 60–80% before.
03Bonus Win

Sharing blocks across requests

Once the cache lives in blocks with a lookup table, identical blocks can be shared instead of duplicated — exactly like copy-on-write pages in an OS. Two big cases:

Shared system prompt → one copy, many requests

When many requests share a long system prompt (or a RAG document, or few-shot examples), vLLM's automatic prefix caching stores those blocks once and points every request at them. Parallel samples from one prompt share the prompt's blocks until they diverge, then copy-on-write only the block that changes.

Why it compounds. Less memory per request → more requests fit at once → bigger batches → the GPU's tensor cores stay busy → higher throughput. Memory efficiency and speed are the same lever here.
04The Scheduler

Continuous batching keeps the GPU full

Flexible block memory unlocks one more win. Static batching groups N requests, runs them together, and waits for the slowest to finish before starting the next batch — so finished sequences leave the GPU idle. Continuous batching slots a new request into a free spot the instant one finishes. Compare:

PagedAttention turns KV-cache memory from the bottleneck that caps batch size into a non-issue, so a single GPU can serve far more concurrent users. It's why vLLM became the default open-source serving engine, and the technique now appears in nearly every production inference stack.

Source: Kwon et al., "Efficient Memory Management for LLM Serving with PagedAttention," SOSP 2023 (arXiv:2309.06180), and the vLLM project.
The trade-offs PagedAttention introduces ▾

Nothing is free. Paging adds a custom attention kernel that must gather K/V from non-contiguous blocks (a little slower per-op than a dense read), CPU bookkeeping to track thousands of block tables, and a preemption/swapping path for when the block pool runs out — evicting a sequence's blocks to CPU RAM and recomputing or swapping them back later, which causes latency spikes. In practice the memory win dwarfs these costs, which is why it's near-universal.
