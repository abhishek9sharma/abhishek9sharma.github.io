Build an LLM · Step 7 — Serving, part 1

The KV cache: why generation isn't quadratically slow

In Step 6 you saw generation is a loop — predict one token, append, repeat. Done naively, every step would recompute attention over the entire history from scratch. The KV cache is the one trick that makes serving LLMs practical: store each token's Keys and Values once, reuse them forever.

Builds directly on Step 3 (attention: Q·K, then ×V) and Step 6 (the generation loop). Self-contained — a quick recap is included.
00Concrete Example First

Matrix multiplication, step by step — with and without cache

The KV cache insight lives inside the attention formula softmax(QKT/√d) · V. Let's trace it with real numbers for "The dog sat", watching exactly which matrix operations change when we add a cache.

Without KV cache — full matrix multiply every step

Choose how many tokens are in context. Walk through all 4 sub-steps. Notice the shapes grow — and that most of the work is wasted.

With KV cache — Q shrinks to one row, shapes change completely

Same 3-token step but now K and V come from cache. Q is just one row (the new token only). The matrix multiply becomes a vector × matrix — fundamentally cheaper.

Why this works — three facts in order

1. Only the last output row predicts the next token. The transformer computes one output vector per token, but the prediction head only reads the last one. All other output rows are thrown away immediately.
2. Therefore only the last Q is needed. The last output row = last Q dotted against all Ks. The other rows of Q only produce the output vectors we discard. We never need old Qs during inference.
3. Therefore all Ks and Vs must be kept — but computed only once. The single new Q attends over every past K and V. They never change after being computed. Store them once, reuse them every step. That is the KV cache.
01The Redundancy

Without a cache, you redo everything every step

Recall from Step 3: to process a token, the model computes its Query and dots it against the Key of every earlier token, then blends their Values. Here's the catch during generation — the Keys and Values of past tokens never change. Yet a naive loop recomputes all of them at every step:

Generating token by token — naive (no cache)

The waste compounds. At step n, the naive approach recomputes K and V for all n tokens. Summed across a sequence that's 1+2+3+…+n ≈ n²/2 Key/Value computations — quadratic in sequence length, almost all of it redundant.
02The Fix

Cache the Keys and Values

The fix is almost embarrassingly simple: after computing a token's K and V, keep them. On the next step you only compute K and V for the single new token and append to the cache. The Query is only ever needed for the current token. Watch the cache grow by exactly one column per step:

Generating token by token — with KV cache

Why only K and V — not Q? A cached token never needs its Query again: it already "asked its question" on the step it was generated. But it remains an answer source for all future tokens, so its Key (how it's matched) and Value (what it contributes) must stick around. Hence KV cache, not QKV cache.
03The Payoff

Quadratic work becomes linear

Per generated token, the cached version does a fixed amount of new K/V work; the naive version does work proportional to the whole history. Across a full sequence the difference is dramatic:

The cache trades compute for memory. You stop recomputing — but now you must store all those Keys and Values. For long contexts and many concurrent users, that storage becomes the dominant cost, and the central problem of LLM serving. Let's size it.

05Two Phases

Prefill vs. decode — the two stages of every generation

The KV cache splits inference into two very different phases. Understanding them explains why the first token is slow and all subsequent tokens are fast.

Phase 1 — Prefill

When you send a prompt, the model processes all input tokens in parallel in a single forward pass. This is fast because modern GPUs excel at matrix × matrix operations. The output: K and V for every prompt token are computed and stored in the cache, and the first output token is generated.

Prompt: "The dog sat on the" → 5 tokens processed together
Q [5×d] × Kᵀ [d×5] = score [5×5] → last row only → first token
✓ Cache now holds K,V for all 5 prompt tokens
This is the pause before ChatGPT starts typing. You see the spinner, then suddenly the first word appears. That pause is the prefill phase: processing your entire prompt, building the cache, generating the first token.

Phase 2 — Decode

Every token after the first is generated one at a time using the cache. Each decode step is:

q [1×d] × K_cache [d×n] = score [1×n] → softmax → × V_cache → output → next token
n grows by 1 each step. Q is always 1 row. Vector-matrix multiply, not matrix-matrix.
This is why streaming feels fast. Once the first token appears, each subsequent token only needs one new K,V computation plus a vector-matrix multiply. The work per token stays nearly constant regardless of how long the output grows.

Where this is heading

So the KV cache is essential and huge — and its size per request grows and shrinks unpredictably as each user's generation runs. Allocating that memory naively wastes 60–80% of it to fragmentation. The fix borrows a 1960s operating-systems idea — paging — and is what makes vLLM fast. That's Step 8.
